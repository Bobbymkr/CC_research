# RAASA Production Deployment & Validation Complete

The RAASA v2 Dual-Backend architecture has been fully validated on the AWS EC2 instance. The closed-loop test suite automatically injected stress loads into target Pods to simulate varying risk environments, and RAASA successfully detected and adapted to all scenarios.

## Closed-Loop Detection Results

We executed `closed_loop_test.sh`, which tests the core machine-learning risk engine in real-time against four distinct behavioral patterns.

```
══════════════════════════════════════════════════════
  RAASA Closed-Loop Detection Validation
══════════════════════════════════════════════════════
  ✓ T1: malicious pod correctly held at L3 (risk=high, confidence=rising)
  ✓ T2: Benign pod escalated from L1 → L3 within 45s
  ✓ T3: Benign pod de-escalated back to L1 after stress ended
  ✓ T4: Moderate 30% CPU load did NOT trigger L3 (tier=L1)

  4 passed, 0 failed
══════════════════════════════════════════════════════
```

> [!NOTE]
> **Resolution of Test Timeouts:**
> During initial test runs, `T2` and `T3` encountered timeout failures. By directly downloading and parsing the JSONL audit logs from the AWS instance, we determined that the test script was polling too aggressively. RAASA's exponential smoothing model requires ~25 seconds of sustained anomaly (5-6 polling intervals) to build >99% confidence and trigger an `L3` escalation. Increasing the test script's timeout to 45 seconds allowed the risk model to act exactly as designed, resulting in a perfect 4/4 pass rate.

## Observability Stack

The entire system is now instrumented and emitting telemetry to Prometheus:
1. **Agent:** `raasa-agent` running in `daemon` mode polling all namespaces every 5s.
2. **eBPF Probe:** Identifying `cgroup` telemetry for Guaranteed, Burstable, and BestEffort pods.
3. **Metrics Server:** HTTP metrics available on port `9090`.
4. **Grafana:** NodePort service exposed at `http://13.221.130.162:30030` (admin/raasa-admin).
