## RM_practical

This folder is a paper-writing companion for the RAASA project. The goal is simple: someone should be able to use only this folder to understand the problem, architecture, implementation, experimental story, evidence, limitations, and future direction of the project well enough to write a defensible research paper.

### What each folder is for

- Note: this folder already contained some earlier loose artifacts at the top level before this curated pass. The canonical packet is the structured layout described below.

- `paper_draft/`  
  The draft paper that was already in the repo, plus a plain-text extraction for quick search.

- `docs/`  
  Existing project notes, evolution writeups, threat model notes, tuning notes, and roadmap material copied from the main repo.

- `configs/`  
  YAML configurations used by the controller, including the tuned linear profile that currently has the strongest evaluation story.

- `code/`  
  Only the essential implementation files needed to explain the architecture, control loop, workloads, experiments, analysis, and Kubernetes path.

- `results/`  
  Representative raw logs (`.jsonl`) and summary outputs (`.json`) for the main evaluation story: static baselines, adaptive linear runs, ML ablation, medium/large scale, syscall-enriched evaluation, older cloud-native/K8s evidence, and the latest April 26 v2 operational validation slice through Phase 1D universal pod-resolution validation.

- `figures/`  
  Paper-ready plots already generated by the project.

- `verification/`  
  Unit test source files that show which parts of the system were intended to be regression-checked.

- `environment/`  
  Reproducibility support files such as `requirements.txt`, `pyproject.toml`, `pytest.ini`, and the trained Isolation Forest artifact.

### Most important caution

The draft paper is broader and more ambitious than the strongest repo-backed implementation claims. Use `06_paper_alignment_and_claim_boundaries.md` before finalizing any paper text. That file separates:

- what is clearly implemented in this repo,
- what is partially implemented or prototype-only,
- what reads more like planned future work than current evidence.
