# Abstract
Modern cloud platforms run many different workloads from many different customers on the same physical machines. This model — called multi-tenancy — keeps costs low and resource use high, but it creates a fundamental security problem: if one tenant's workload behaves maliciously, it may be able to harm or spy on a neighboring tenant's workload that shares the same physical host.
The tools that currently protect these workloads use fixed, unchanging security rules. A container starts with a security policy and keeps that same policy for its entire lifetime, regardless of whether its behavior changes. This creates a well-known trade-off: strong security rules slow down workloads that do not need them, while weak security rules leave genuinely dangerous workloads poorly protected.
This paper proposes RAASA (Risk-Aware Adaptive Sandbox Allocation), a framework that continuously watches each running container and computes a real-time risk score ρ ∈ [0,1] from four behavioral signals: system call patterns, file access entropy, network traffic anomalies, and CPU usage bursts. Based on this score, RAASA automatically assigns each container to one of three isolation tiers (L1–L3) and enforces the corresponding policy — through seccomp filter updates, cgroups v2 resource limits, and network namespace rules — without restarting the workload.
RAASA fills a gap that no existing tool addresses. Detection tools such as Falco identify threats but respond only by killing the container or sending an alert. Strong isolation runtimes such as Kata Containers and gVisor provide excellent protection but apply the same isolation level regardless of observed risk. The recently published eBPF-PATROL system (November 2025) provides adaptive eBPF-based enforcement but uses binary block/allow decisions with no quantitative risk scoring and no graduated tier system. RAASA is the first framework to combine continuous quantitative risk scoring with a graduated, dynamically enforced isolation spectrum — occupying the previously unoccupied space between lightweight monitoring and rigid strong isolation.

# 1. Introduction
Cloud computing now powers almost every type of modern digital service — from web applications and databases to machine learning model inference and real-time IoT backends. The economic model that makes cloud computing affordable depends on multi-tenancy: many customers share the same physical hardware, with the cloud provider using software to keep their workloads separated. This sharing allows providers to charge less per user while keeping machines running at high utilization.
The isolation between tenants is enforced by a combination of hardware virtualization (hypervisors), operating system features (Linux namespaces and control groups), and software-defined networking. These mechanisms work well under normal conditions, but they have well-documented weaknesses. A container breakout exploit, a microarchitectural side-channel attack such as Spectre or Meltdown, or a misconfigured network segment can create a path from one tenant's workload to another's data. These are not theoretical concerns: container escape vulnerabilities such as runc CVE-2019-5736 have been actively exploited in real deployments.
Cloud operators must therefore balance two objectives that pull in opposite directions. They want isolation strong enough to stop real attacks, but they also want their platforms to be fast and efficient. Applying the strongest possible isolation to every container — for example, running each one inside a dedicated microVM as Kata Containers does — costs between 300 and 600 milliseconds of startup time and 50 to 100 MB of extra memory per instance [10]. For the vast majority of workloads that are completely benign, this overhead is wasted. On the other hand, applying the weakest isolation to every container leaves genuinely risky workloads inadequately protected.
The key insight behind RAASA is that this trade-off is false: it only exists because current systems treat isolation as a fixed, binary choice made at container creation time. If the isolation level could adapt continuously based on measured evidence of risk, benign workloads could run with low overhead while suspicious workloads receive stronger protection — automatically, in real time, without human intervention.
This paper makes the following contributions. First, it presents a formal behavioral risk model for containerized workloads, defining a five-feature vector and a weighted scoring function that produces a normalized risk score ρ ∈ [0,1]. Second, it defines a three-tier isolation policy framework (L1–L3) and the precise enforcement mechanisms that implement each tier using standard Linux kernel interfaces. Third, it presents RAASA, a closed-loop system that continuously connects risk scoring to policy enforcement without workload restart. Fourth, it provides a critical analysis of the seccomp filter relaxation problem and proposes a principled solution. Fifth, it defines a rigorous evaluation plan with a priori success criteria including weight sensitivity analysis.

# 2. Security Challenges in Multi-Tenant Cloud Environments
The shared-resource model of multi-tenant cloud platforms creates an environment where any incomplete isolation boundary is a potential attack path. When many tenant workloads run on a single physical host — sharing CPU caches, memory channels, network interfaces, and storage controllers — an attacker who partially breaks out of their container gains access to resources that were never intended to be reachable.
## 2.1 Cross-Tenant Attack Vectors
The main threat categories in multi-tenant cloud environments are as follows.
- Co-residency attacks: An attacker deliberately arranges for their malicious workload to be scheduled on the same physical host as a target workload, either by exploiting knowledge of the scheduler's placement algorithm or by running many instances until co-location occurs by chance [2].
- Microarchitectural side-channel attacks: Shared CPU hardware — particularly the Last-Level Cache (LLC), Translation Lookaside Buffer (TLB), and branch predictor — can be observed by a co-resident attacker to infer memory access patterns of a victim workload. The Spectre and Meltdown vulnerability classes demonstrated that these attacks can leak cryptographic keys and other secrets across isolation boundaries.
- Container escape exploits: Vulnerabilities in the Linux kernel or in the container runtime itself can allow a workload to break out of its namespace isolation and gain access to the host operating system. The runc CVE-2019-5736 vulnerability is one documented example that affected production deployments across major cloud providers.
- Lateral movement: An attacker who has compromised one container may pivot through shared network namespaces, mounted volumes, or inter-process communication channels to reach adjacent tenant containers on the same host.

These threats are made worse by the dynamic and short-lived nature of cloud workloads. Autoscaling, serverless function invocations, and microservice orchestration through systems like Kubernetes create workload placement patterns that change rapidly and unpredictably. Static threat perimeter definitions cannot keep pace with this dynamism.
## 2.2 Threat Model
RAASA operates under the following threat model. The attacker is assumed to have already achieved code execution within at least one tenant container — either by exploiting a vulnerability in the containerized application (for example, a remote code execution flaw in an unpatched web service) or by deliberately deploying a malicious workload as a tenant. The attacker's goals are one or more of the following: (a) exfiltrating sensitive data from co-resident tenant workloads via side-channel observation or direct filesystem access; (b) escalating privileges to the host kernel via a container escape exploit; or (c) spreading malicious activity laterally to adjacent containers within the same multi-tenant environment.
The following are explicitly out of scope: attacks that require physical access to hardware, cryptographic breaks of tenant data at rest, supply-chain attacks on the container image registry, and attacks that exploit vulnerabilities in the host hypervisor firmware. RAASA's containment model is specifically designed to detect and restrict anomalous in-container runtime behavior that is indicative of threats (a) through (c).

# 3. Limitations of Existing Sandboxing Approaches
## 3.1 Static Containment Policies
Existing container sandboxing mechanisms — seccomp system call filtering, AppArmor and SELinux mandatory access control profiles, cgroups v2 resource controllers, and Linux capabilities restrictions — are all configured statically at container launch time. This design assumes that the risk profile of a workload is uniform and predictable across its entire execution lifetime. In practice, this assumption is wrong in two important ways.
First, application behavior changes across execution phases. A containerized web service may need to perform privileged filesystem operations during startup — mounting configuration volumes, binding to low-numbered ports — that are completely unnecessary, and potentially dangerous if permitted, during its normal request-handling phase. A static seccomp profile that permits these calls throughout execution keeps the attack surface unnecessarily wide during the long steady-state period of operation.
Second, workload risk is not binary. The trusted/untrusted distinction that static policies enforce fails to capture the continuous spectrum of behavioral change that typically precedes or accompanies an attack. A workload that begins making an unusually high volume of ptrace() system calls, or that suddenly opens a raw socket it did not use during its profiling phase, is exhibiting elevated risk. A static policy must either permit or deny these calls unconditionally, with no intermediate response available.
## 3.2 Performance Cost of Uniform Strong Isolation
Applying maximum isolation uniformly to all workloads — for example, running every container inside a microVM using Kata Containers or Firecracker — imposes measurable performance costs that are unjustifiable for the majority of workloads. Kata Containers launches a dedicated guest kernel per pod, introducing cold-start latencies of 300–600 ms and per-instance memory overhead of 50–100 MB [10]. Google's gVisor interposes a user-space kernel (the Sentry) between the application and the host kernel, adding latency on I/O-intensive and syscall-intensive workloads. For a deployment running thousands of short-lived, benign microservice containers, these costs accumulate to a significant efficiency penalty. Static strong isolation sacrifices the performance benefits that containerization was designed to provide.

# 4. Related Work and Research Gap
## 4.1 Runtime Anomaly Detection Tools
Falco (CNCF) is the most widely deployed runtime security tool for containerized environments [11]. It uses eBPF-based kernel probes to intercept system calls and evaluates them against a rule engine, generating alerts when predefined patterns are matched — for example, when a shell is spawned inside a container or a sensitive file is read. Sysdig Secure extends Falco's detection engine with behavioral profiling. However, both tools are fundamentally alert-generation systems: they detect and report anomalous behavior but do not modify the container's isolation posture in response. Falco's enforcement options are limited to logging an event, sending a webhook notification, or terminating the container via SIGKILL. There is no concept of a graduated, intermediate containment response.
## 4.2 Strong Isolation Runtimes
Kata Containers and Firecracker provide VM-level isolation for OCI-compatible container workloads by running each pod inside a lightweight microVM with a dedicated guest kernel and hardware-virtualized memory [10]. This eliminates the shared-kernel attack surface entirely. gVisor interposes a user-space kernel between the application and the host kernel, substantially reducing the host kernel attack surface [9]. Crucially, neither class of tool adapts its isolation level at runtime based on observed workload behavior — isolation is fixed at container creation and remains constant regardless of how the workload behaves during execution.
## 4.3 Tetragon (Cilium) — Production eBPF Enforcement
Tetragon, developed by the Cilium project and now a CNCF sandbox project, is the most production-mature eBPF-based enforcement tool available [17]. It provides real-time security observability and runtime policy enforcement at the kernel level, is Kubernetes-aware — understanding namespaces, pod identities, and workload metadata — and is actively deployed in production environments at scale.
Tetragon shares RAASA's syscall interception capability and requires no workload restart. However, its policies are defined as static TracingPolicy YAML rules authored before deployment. Tetragon has no mechanism for computing a continuous risk score from telemetry, no concept of graduated isolation tiers, and no ability to dynamically reconfigure cgroup resource controllers or network namespace rules in response to observed behavioral change. Like Falco, it is a policy enforcement tool — not an adaptive risk-scoring system. RAASA can complement Tetragon: a confirmed Tetragon policy violation can be fed directly into RAASA's risk estimator as a high-confidence signal, immediately elevating ρ toward the L3 threshold.
## 4.4 eBPF-PATROL (2025) — The Closest Prior Work
The most recent and closely related work is eBPF-PATROL (arXiv, November 2025), which explicitly addresses the limitation that existing approaches such as seccomp and MAC frameworks lack context awareness and adaptive enforcement. eBPF-PATROL uses eBPF probes to monitor system calls, maintains per-process and per-container behavioral profiles through an Event Analyzer module, and can block specific attack behaviors — reverse shells, container escape attempts, and capability abuse — in real time, with reported overhead below 2.5%.
RAASA differs from eBPF-PATROL in three fundamental ways. First, eBPF-PATROL's enforcement decisions are binary: a system call is either blocked or permitted. RAASA computes a continuous, quantitative risk score ρ ∈ [0,1] that captures the degree of behavioral deviation from a workload's baseline, not just the presence or absence of a specific attack pattern. Second, eBPF-PATROL does not define or enforce graduated isolation tiers — it has no concept of L1, L2, and L3 containment levels that map risk scores to specific combinations of seccomp policy, cgroup resource limits, and network namespace configuration. Third, eBPF-PATROL does not dynamically reconfigure cgroup resource controllers or network namespace policies in response to a behavioral signal. RAASA's graduated, multi-dimensional policy enforcement loop — driven by a quantitative risk score — is not provided by eBPF-PATROL or any other existing tool.

Table 1 summarizes the capabilities of the most relevant existing tools against RAASA's design objectives, now including both eBPF-PATROL and Tetragon as the two closest prior works. The critical gap that remains is the absence of any system that: (i) computes a continuous, quantitative risk score from multiple behavioral signals; (ii) maps that score to a graduated multi-dimensional isolation policy covering seccomp, cgroups, and network namespaces simultaneously; and (iii) enforces that policy dynamically without workload restart. RAASA is designed to occupy precisely this gap.

# 5. The RAASA Framework
## 5.1 System Architecture
RAASA implements a closed-loop, continuous control architecture comprising five interconnected modules. The control loop executes at a configurable polling interval — defaulting to five seconds — for each active container, producing a risk score and, where necessary, a sandbox policy transition. The five stages are: (1) Telemetry Collector, (2) Feature Extractor, (3) Risk Estimator, (4) Policy Selector, and (5) Container Runtime Enforcer. Data flows from stage (1) through to stage (5), with the enforcement at stage (5) feeding back implicitly into stage (1) through the changed behavioral constraints applied to the workload.
## 5.2 Telemetry Collection
The Telemetry Collector instruments each containerized workload at the host kernel level using two complementary mechanisms. First, the Linux audit subsystem (auditd) captures all system call invocations with argument metadata, filtered per-container using audit rules keyed on cgroup membership. Second, the cgroups v2 statistics interfaces (cpu.stat, io.stat, memory.stat) are polled at each control loop iteration to obtain CPU throttling events, block I/O operation counts, and resident set size. Network telemetry is collected via /proc/net/dev and iptables packet counters, capturing inter-container and external egress byte volumes. This host-level collection approach requires no modification to the monitored workload image, preserving transparency to the tenant.
An initial profiling phase of 120 seconds establishes a per-workload behavioral baseline immediately after container start. The robustness of this baseline against profile poisoning — where an attacker behaves benignly during profiling and attacks afterward — is addressed in Section 5.6 (Limitations and Mitigations).
## 5.3 Feature Extraction
The Feature Extractor transforms raw telemetry into a normalized feature vector F = [f1, f2, f3, f4, f5] over each collection window W (default: 30 seconds).
- f1 — Syscall Frequency Anomaly Score: The observed system call distribution over W is compared against the per-workload baseline profile using Jensen-Shannon Divergence (JSD). JSD is chosen because, unlike KL divergence, it is symmetric and always finite, producing a value in [0,1] where 0 indicates an identical distribution to baseline and 1 indicates complete divergence.
- f2 — Privileged Syscall Rate: The rate of system calls from a predefined high-risk watchlist — including ptrace, clone with CLONE_NEWUSER, mknod, mount, setuid, and socket with AF_RAW — normalized against the total system call rate in the window.
- f3 — File I/O Entropy: The Shannon entropy of file path access patterns, computed over unique file paths accessed in W. High entropy relative to baseline indicates access to atypical filesystem locations, which is a common indicator of reconnaissance or data exfiltration activity.
- f4 — Network Egress Anomaly: The Z-score of outbound byte volume relative to the workload's rolling mean and standard deviation, capped to [0,1] to prevent outlier distortion.
- f5 — CPU Burst Index: The ratio of CPU throttling events (cpu.stat throttled_usec) to total CPU time in W, normalized to [0,1]. Sustained CPU bursting in an otherwise steady-state workload can indicate a fork bomb or cryptomining attack.
## 5.4 Risk Estimation
The Risk Estimator computes a scalar risk score ρ ∈ [0,1] as a weighted linear combination of the feature vector: ρ = w1·f1 + w2·f2 + w3·f3 + w4·f4 + w5·f5, where w1 + w2 + w3 + w4 + w5 = 1.
Default weights are initialized as: w1 = 0.35 (syscall anomaly), w2 = 0.30 (privileged syscall rate), w3 = 0.15 (I/O entropy), w4 = 0.12 (network egress), w5 = 0.08 (CPU burst). The weighting of syscall-based features (w1 + w2 = 0.65) reflects the established primacy of system call behavior as an indicator of malicious activity [13, 14], and is consistent with the feature importance ordering observed in prior anomaly detection literature [15].
These default weights are not fixed design constants — they are starting points subject to empirical validation. The planned evaluation (Section 7) includes a weight sensitivity analysis using a grid search over weight combinations, measuring the variance in ρ and tier classification outcomes. A successful result will demonstrate that the system's behavior is robust to reasonable perturbations of the weight vector, which is a necessary condition for practical deployment.

When a tier transition is required, the Policy Engine applies the new containment configuration using three parallel enforcement mechanisms.
- cgroups v2 resource reconfiguration: CPU quota (cpu.max) and memory limits (memory.max) are updated by writing directly to the container's cgroup v2 hierarchy at /sys/fs/cgroup. This takes effect immediately without process interruption and is the most straightforward of the three enforcement mechanisms.
- Network namespace policy update: iptables rules are dynamically inserted or removed for the container's network namespace, restricting or restoring inter-container and external egress traffic. Both tightening and relaxation of these rules are supported by the iptables API without process restart.
- seccomp filter management: This is the most technically constrained of the three mechanisms. The Linux seccomp filter chain is strictly additive and increasingly restrictive — a process can only add more restrictive filters to its chain, not remove or relax existing ones. This means that a direct hot-swap from an L3 (deny-all) filter to an L1 (permissive) filter within the same process is not possible without a process restart. RAASA handles this constraint through a process-restart-with-state-preservation mechanism for downward tier transitions (L3→L2 or L2→L1), implemented using CRIU (Checkpoint/Restore In Userspace) — a standard Linux tool that freezes a running process, serializes its memory and file-descriptor state to disk, and restores it into a new process instance with the updated seccomp policy applied. For upward transitions (L1→L2 or L2→L3), an additional restrictive filter is appended to the chain, which is fully supported without restart. This asymmetry is an acknowledged limitation and is discussed further in Section 5.6.
## 5.6 Acknowledged Limitations and Mitigations
Transparency about the following limitations is important for an accurate evaluation of RAASA's practical applicability.
- Baseline profiling vulnerability: An attacker who is aware of RAASA's 120-second profiling window could behave normally during this period and launch an attack immediately afterward. RAASA addresses this through two mitigations: (i) the profiling window duration is randomized within a configurable range, making it harder to predict precisely; and (ii) a conservative default baseline is applied before profiling completes, treating any unobserved system call as mildly suspicious. Periodic re-baselining every 30 minutes is also supported to adapt to legitimate long-term behavioral drift.
- seccomp downgrade constraint: As described above, relaxing a seccomp filter requires a process restart. The restart path is fast (sub-100ms for stateless workloads) but is not zero-cost. For stateful workloads, state preservation adds latency. This is a fundamental Linux kernel constraint, not a RAASA design choice, and is shared by any system that attempts to relax seccomp restrictions at runtime.
- Linear scoring model: The weighted linear risk score may underperform on workloads with complex, non-linear behavioral signatures. A feature interaction that is individually low-risk but collectively high-risk (for example, moderate syscall anomaly combined with moderate network egress) may not score accurately under a linear model. Future work will investigate unsupervised anomaly detection models (Isolation Forest, LSTM sequence models) as drop-in replacements for the scoring function.
- Single-host prototype scope: The current prototype and planned evaluation operate on a single Linux host running Docker. Scaling to a Kubernetes cluster requires a per-node RAASA agent architecture with a centralized risk aggregation plane, which is planned as future work.

# 6. Differentiation from Existing Tools
A precise comparison with the three most important existing approaches clarifies RAASA's novel contribution.
Falco detects anomalous behavior and sends alerts or kills containers. It does not modify any isolation parameter — not the seccomp filter, not the cgroup allocation, not the network namespace rules — in response to a risk signal. Its response is binary and destructive. RAASA, by contrast, responds to elevated risk with a proportional, graduated increase in isolation that keeps the workload running while restricting its capabilities.
gVisor and Kata Containers provide strong structural isolation at the cost of higher resource overhead. Their isolation level is fixed at container creation — a Kata container has identical isolation whether its behavioral risk score is 0.02 or 0.98. RAASA is designed to complement these tools, not replace them: Kata Containers can serve as RAASA's L3 enforcement backend for the small fraction of workloads that reach the highest risk tier, while L1 and L2 workloads continue to run as standard containers with lower overhead.
eBPF-PATROL provides binary block/allow enforcement based on specific detected attack patterns. It does not compute a continuous risk score, does not define graduated isolation tiers, and does not reconfigure cgroup or network namespace policies dynamically. The two systems address different but complementary problems: eBPF-PATROL excels at blocking specific known attack behaviors with extremely low overhead; RAASA provides a broader behavioral risk assessment that can elevate isolation in response to suspicious-but-not-yet-confirmed threat activity. RAASA can use confirmed eBPF-PATROL detections as direct inputs to its risk estimator, immediately elevating ρ to the L3 threshold upon a confirmed alert.

# 7. Planned Evaluation
## 7.1 Experimental Setup
Experiments will be conducted on a single Linux host (Ubuntu 22.04 LTS, kernel 5.15+) running Docker Engine 24.x with cgroups v2 enabled. This single-host scope is explicitly chosen as a controlled environment for validating the core control loop; multi-host Kubernetes deployment is deferred to future work and is explicitly acknowledged as a limitation (Section 5.6). Three categories of workloads will be deployed.
- Benign workloads: nginx web server, a Python data processing script, and a PostgreSQL database container — representing standard, well-understood cloud application profiles that should remain at L1 throughout their execution.
- Resource-intensive workloads: A CPU-bound Monte Carlo simulation, a memory-intensive matrix computation, and a network-intensive data streaming process. These are included specifically to test false positive behavior: these workloads are benign but resource-intensive, and a well-calibrated system should not escalate them to L2 or L3.
- Adversarial workloads: Containers executing known attack patterns: (i) abnormal system call sequence injection via a synthetic strace replay tool; (ii) a controlled fork bomb variant; (iii) a simulated container escape attempt using ptrace on sibling processes; (iv) a simulated data exfiltration pattern involving high-entropy file reads followed by unusual network egress.

## 7.3 Baseline Comparison
RAASA will be benchmarked against two baselines. Baseline A will apply a static L1 policy uniformly to all workloads — the current Docker default configuration, representing the efficiency-optimised but security-weak approach. Baseline B will apply a static L3 policy uniformly, representing the security-maximised but efficiency-degrading configuration. The comparison between RAASA and these two baselines will directly quantify both the efficiency cost of over-isolation and the security cost of under-isolation, providing empirical grounding for RAASA's adaptive approach.

# 8. Future Work
The current RAASA design uses a weighted linear scoring model for risk estimation. While computationally efficient and interpretable, this model is likely to underperform on workloads whose behavioral signatures involve non-linear feature interactions. A planned next phase of research will evaluate the integration of unsupervised anomaly detection models — specifically Isolation Forest and LSTM-based sequence models trained on system call n-gram representations — as drop-in replacements for the weighted scoring function, following the same feature extraction pipeline defined in Section 5.3.
Extending RAASA to a distributed Kubernetes cluster environment will require integration with the Container Runtime Interface (CRI) and a per-node RAASA agent architecture coordinated by a centralized risk aggregation plane. Integration with existing service meshes such as Istio and Linkerd for network-layer telemetry enrichment is also planned. Automated seccomp profile generation from workload profiling traces using tools such as oci-seccomp-bpf-hook will eliminate the manual profile authoring step currently required for L2 and L3 tier definitions.
Finally, the seccomp downgrade constraint identified in Section 5.5 motivates investigation of an alternative enforcement architecture based exclusively on BPF LSM programs, which offer dynamic load and unload semantics that could support true bidirectional filter reconfiguration without process restart.

# 9. Conclusion
This paper has presented RAASA, a Risk-Aware Adaptive Sandbox Allocation framework that addresses the fundamental security-efficiency tension in multi-tenant cloud environments. By continuously computing a quantitative risk score from system call distributions, privileged call rates, file I/O entropy, network egress anomalies, and CPU burst patterns, RAASA dynamically assigns each containerized workload to one of three isolation tiers and enforces the corresponding policy through cgroups v2 reconfiguration, network namespace rule updates, and a carefully managed seccomp filter strategy — without workload restart for upward transitions.
A thorough review of the existing literature and production tooling confirms that the gap RAASA addresses remains open. The nearest prior work, eBPF-PATROL (2025), provides valuable binary enforcement of specific attack behaviors but does not implement quantitative risk scoring or graduated policy tiers. Detection tools such as Falco respond only with alerts or termination. Strong isolation runtimes such as Kata Containers and gVisor fix their isolation level at container creation and do not adapt at runtime.
RAASA occupies the previously unoccupied position between lightweight detection and rigid strong isolation, providing a continuous, quantitative containment spectrum that no existing tool offers. The planned evaluation is designed to empirically validate RAASA's detection accuracy, containment effectiveness, and performance overhead against both static-policy baselines and the latest existing tools, with all success criteria defined before experiments are conducted.
